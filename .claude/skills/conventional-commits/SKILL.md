# conventional-commits

Форматирует коммит-сообщения строго по спецификации [Conventional Commits](https://www.conventionalcommits.org/). Только формат — git-операции выполняет `ship-pr`.

## Формат

```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

- **description**: imperative, строчные (lowercase), без точки в конце
- **scope**: опционален, в скобках после type
- **body**: опционален, отделяется пустой строкой, объясняет ЧТО и ПОЧЕМУ
- **footer**: опционален, `BREAKING CHANGE:` или референсы (напр. `Closes #123`)

## Типы

**Главное правило**: `feat`/`fix`/`perf` — только если изменение видно конечному пользователю приложения. Внутренний тулинг, настройка репо, CI, deps — всегда `chore`.

| Тип        | Когда использовать                                                                                                         |
| ---------- | -------------------------------------------------------------------------------------------------------------------------- |
| `feat`     | Новая фича для **пользователя** продукта (минорная версия)                                                                 |
| `fix`      | Исправление бага, видимого **пользователю** (патч-версия)                                                                  |
| `docs`     | Только документация — README, комментарии, спека                                                                           |
| `style`    | Форматирование, пробелы, точки с запятой — без изменения логики                                                            |
| `refactor` | Изменение кода без фич и багфиксов                                                                                         |
| `perf`     | Оптимизация производительности, заметная **пользователю**                                                                  |
| `test`     | Добавление или исправление тестов                                                                                          |
| `build`    | Изменение сборки или внешних зависимостей                                                                                  |
| `ci`       | Изменение CI/CD-конфигурации                                                                                               |
| `chore`    | Внутреннее обслуживание: обновление deps, настройка тулинга, CLI-скиллы, конфиги репо — всё, что **не видно пользователю** |
| `revert`   | Откат предыдущего коммита                                                                                                  |

## Breaking changes

Два способа обозначить ломающее изменение:

1. `!` перед `:` — `feat(api)!: убрать поле name из ответа`
2. Футер `BREAKING CHANGE:` с описанием

```
feat(api)!: убрать поле name из ответа

BREAKING CHANGE: поле name удалено из UserResponse, используй firstName + lastName
```

## Процесс

1. **Определить type**. Если из того, что меняется, неочевидно — спросить пользователя через AskUserQuestion, предложив 2-4 релевантных варианта.
2. **Определить scope** (опционально). Обычно по имени пакета или модуля. Если неясно — спросить или пропустить.
3. **Сформулировать description**: глагол в императиве (добавить, исправить, обновить), строчные, без точки. Уложиться в 72 символа.
4. **Добавить body** только если изменение требует объяснения (почему, а не что).
5. **Добавить footer** для breaking changes или референсов на issues.

## Примеры

```
feat(frontend): добавить страницу списка бронирований
```

```
fix(api): поправить фильтрацию прошедших слотов
```

```
refactor(frontend): вынести хуки бронирования в отдельный файл
```

```
feat(spec)!: переименовать Appointment в Booking

BREAKING CHANGE: все эндпоинты /appointments теперь /bookings
```

## Интеграция с ship-pr

`ship-pr` выполняет git-операции (stage, commit, push, PR), но использует свободный формат коммитов. Если пользователь вызвал `conventional-commits` и затем хочет закоммитить — отформатировать сообщение по CC. Если пользователь вызвал `ship-pr` с явным запросом «конвенциональный коммит» — применить правила этого навыка.
